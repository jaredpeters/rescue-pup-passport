# CLAUDE.md — project context for AI assistants

Use this file to bootstrap yourself when helping with Rescue Pup Passport. Read [README.md](./README.md) and [SETUP.md](./SETUP.md) for user-facing detail.

## What this project is

An open-source web app for dog rescuers/fosters to track multiple dogs through rehab to adoption and produce a printable adopter report at the end. Built for non-technical users — ease of setup and forgiving UI matter more than feature density.

## Architecture (one paragraph)

Next.js 16 with `output: "export"` — the entire app is static HTML/CSS/JS. All persistence goes through Supabase Postgres via `@supabase/supabase-js` using the anon key directly from the browser, protected by Row-Level Security. No API routes, no server runtime. `localStorage` is used **only** to remember which dog was last selected (`rpp-selected-dog-id`) — it is not a data store.

## Key files

- `src/lib/types.ts` — the domain model. `Dog` is the profile; `DogData` is the per-dog children (weights, health logs, etc.); `PuppyData = DogData + { profile: Dog }` is the composite shape components consume. `SeedDog` is `DogData + { profile: Omit<Dog, "id"> }` for pre-insert fixtures.
- `src/lib/supabase.ts` — creates the singleton Supabase client. `isSupabaseConfigured()` rejects placeholder values from `.env.local.example`.
- `src/lib/db.ts` — all Supabase queries. `db.dogs.*` for profile CRUD, `db.{weights,healthLogs,...}` for child-row mutations. `loadDogData(dogId)` is the fan-out fetch. `seedIfEmpty()` inserts the three demo dogs on first connect.
- `src/lib/storage.ts` — `getSeedDogs()` returns the three demo profiles, `generateId()`, JSON export/import, CSV import, and the selected-dog-id localStorage pointer. Nothing else should touch localStorage.
- `src/lib/useDogStore.ts` — the single React hook every page uses. Returns profiles, selected dog, full composed `PuppyData`, and all mutators. Optimistic updates with temp ID → real UUID swap.
- `supabase/schema.sql` — the full DDL. Ship it and users run it once. Idempotent (all `if not exists`).
- `supabase/migrations/` — small additive migrations for users who ran an older `schema.sql`.
- `src/components/SetupScreen.tsx` — shown when env vars are missing or placeholder. This is what non-technical users see first if they miss a step.

## Conventions

- **Table names** start with `puppy_` for historical reasons even though the app handles adult dogs. Don't rename without a migration strategy — existing deployments would break.
- **Camel/snake mapping** is done in `db.ts` via `toCamel` / `toSnake`. TypeScript is camelCase, Postgres is snake_case. Don't leak snake_case into React components.
- **IDs** are UUIDs generated by Postgres for real rows. Seed dogs use static string IDs so tests can assert them. `generateId()` is used only for optimistic temp IDs before the DB assigns a real UUID.
- **Dates** are ISO `YYYY-MM-DD` strings everywhere in TS. Postgres `date` type. Never use JS Date in application code except for display formatting.
- **Severity/category/status** are string literal unions mirrored by SQL `check` constraints. Add a value in both places or the insert will fail silently (error is logged, UI shows stale row).

## Testing

Vitest + jsdom. Tests are fast, deterministic, and **do not hit a real Supabase**. The three test files:

- `storage.test.ts` — seed dog integrity, ID utilities, import/export round-trips.
- `db.test.ts` — `isSupabaseConfigured()` behavior (incl. placeholder rejection).
- `supabase-schema.test.ts` — parses `schema.sql` as text and asserts required tables, columns, RLS policies, check constraints. This is the guard against schema drift.

If you add a new column, add both a migration in `supabase/migrations/` **and** an assertion in `supabase-schema.test.ts`.

## What NOT to do

- **Don't add a Node/server runtime.** Static export is a feature — it means any static host works, including GitHub Pages and S3.
- **Don't reintroduce localStorage as a data store.** It was removed because data loss on tab close was the #1 user complaint.
- **Don't hardcode a Supabase URL.** The repo is public; users bring their own. The setup screen and `isSupabaseConfigured()` are the guard.
- **Don't add location-specific content** (vet listings, local shelters, etc). The project serves users anywhere. Keep references generic (AVMA, Cornell, ASPCA, AAHA).
- **Don't add analytics or tracking.** Part of the value proposition is "your data, your database, no third parties."

## Running things

```bash
npm install
npm run dev       # localhost:3000
npm test          # vitest
npm run build     # produces /out for static deploy
```

## When the user is non-technical

The primary user is not a software engineer. When in doubt:

- Prefer editing an existing component over introducing a new concept.
- Write error messages that say what to do, not what went wrong technically.
- Test flows on a fresh Supabase project, not just your own dev database.
